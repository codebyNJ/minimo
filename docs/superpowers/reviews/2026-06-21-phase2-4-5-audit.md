# Audit: Phase 2 (OpenCode), Phase 4 (Export), Phase 5 (Custom Providers)

**Date:** 2026-06-21
**Scope:** Code review + manual test battery against the 9 implementation commits + 2 docs commits covering the OpenCode provider, `ctx export`, and the YAML-based custom provider system.
**Plans audited:** `docs/superpowers/plans/2026-06-21-ctx-phase2-opencode-provider.md`, `2026-06-21-ctx-phase4-export.md`, `2026-06-21-ctx-phase5-custom-providers.md`
**Git range reviewed:** `669a04c` (end of Phase 1) → `3ad4467` (end of Phase 2/4/5 implementation, before fixes)
**Fix commits:** `7810951`, `0fb1b57`

## Method

Two independent passes ran in parallel:
1. A `general-purpose` subagent did a read-only code review against the diff and the three plan documents, with no visibility into the implementer's session history (precisely-crafted context only — per `superpowers:requesting-code-review`).
2. Manual black-box testing of the built `ctx.exe` against simple and moderate cases a common user would hit — CLI argument handling, both providers, export with/without `--with-content`, custom-provider YAML edge cases, config edge cases.

## Code Review Findings

### Strengths (reviewer's words)

- Clean adherence to the Phase 1 `Provider` interface — all three new providers (`opencode`, `configprovider`) implement `Name/Detect/ListSessions/ReadContext` exactly, consumed uniformly by the engine.
- `SessionContext.Files` was added in a correctly backward-compatible way (zero value `nil`, existing struct literals untouched, `omitempty` keeps it out of JSON when empty).
- The privacy gate was structurally right where it counts: `FileRef` stayed paths-only at collection time; content reading happens only at export time behind `--with-content` — sensitive bytes never sit in the in-memory state store.
- OpenCode liveness is honestly constrained — `statusFor` can only return `Idle`/`Ended`, never `Active`, matching the confirmed absence of a live-PID registry.
- The Windows SQLite URI gotcha (`filepath.ToSlash` for `file:` DSNs) is handled, with exactly one comment on the one non-obvious line.
- `configprovider`'s token-SUM recomputes from scratch every tick rather than accumulating into persistent state, which makes it idempotent across re-reads (no double-counting) — confirmed correct, despite the plan having flagged this as a risk to watch.
- Config loading degrades gracefully: missing `config.yaml` → `Default()`; empty `enabled_providers` → "all" — both absent-tolerant as designed.

### Issues — Critical (Must Fix)

None found.

### Issues — Important (Should Fix)

1. **`--with-content` embedded arbitrary out-of-project file content.** `internal/export/export.go` read every tracked path with `os.ReadFile`, with no scoping to the session's working directory. Verified in testing: real exports included `C:\Users\sst\.claude\sessions\14580.json` and a `.gemini\…\logs.json` — both outside the project, both potentially credential-adjacent, exactly the threat model `Architecture.md#8` was written to prevent.
   **Status: FIXED** (commit `7810951`) — content is now only read for paths under `ctx.Session.CWD` (case-insensitive prefix match, since Windows drive-letter casing varies between session cwd and tracked paths). Out-of-project paths still appear in the `files[]` list with their path, but `content` stays empty.

2. **`configprovider` read each matched file twice per refresh tick.** `ListSessions` → `sessionInfo` did a full `os.ReadFile` just to extract a label; `ReadContext` → `sessionInfo` did a second full read of the same file, then two more full-content regex scans. On a 2-second poll tick this is the dominant cost for any non-trivial log file.
   **Status: FIXED** (commit `7810951`) — `ListSessions` now returns IDs only (file path), since the engine always follows up with `ReadContext` for the actual detail. One read per file per tick instead of two.

3. **`*sql.DB` cached for OpenCode's lifetime is never closed; no reconnect if the DB file's identity changes mid-`--watch`.** Low-probability, not a blocker — process exit reclaims the handle for one-shot `ctx status`/`ctx export`; for long-running `--watch`, a deleted-and-recreated `opencode.db` would keep serving the old handle.
   **Status: Accepted as a known, documented tradeoff — not fixed.** Revisit if `--watch` sessions are observed spanning an OpenCode reinstall/migration.

4. **Go version floor inconsistency.** All three plans state "Go 1.23+ floor"; `go.mod` declares `go 1.25.0`. Root cause confirmed by inspecting the module cache: `modernc.org/sqlite@v1.53.0` and its `modernc.org/libc` dependency both declare `go 1.25.0`, which forces the importing module's floor up (Go 1.21+ toolchain rule).
   **Status: Open — needs your call.** Not silently resolved either direction. Options: (a) leave at 1.25 (fine if distribution is prebuilt binaries via goreleaser, per Phase 6), or (b) downgrade `modernc.org/sqlite` to a pre-1.25-floor version to restore the stated 1.23 floor.

### Issues — Minor (Nice to Have)

5. **Zero-time timestamps exported as `0001-01-01T00:00:00Z`.** `epochMillis(0)` returns the Go zero `time.Time`, which `time.RFC3339`-formats to that string rather than being empty.
   **Status: FIXED** (commit `7810951`) — `export.Build` now omits `startedAt`/`lastActive` entirely (`omitempty`) when the time `IsZero()`.

6. **`sessionRow.cost` scanned from OpenCode's `session` table but never surfaced anywhere** (not in the status table, not in export). Harmless dead data today; cost is OpenCode's one genuinely unique, exact metric and is worth surfacing in a future pass.
   **Status: Left as-is**, intentionally — out of scope for this round.

7. **Custom providers register unconditionally in `main()`; correctness depends entirely on `engine.Refresh()`'s `Detect()` gate to suppress no-match providers.** Confirmed correct as designed, just noting the dependency.
   **Status: No action — working as intended.**

8. **Typo'd flags fail silently rather than erroring** (e.g. `ctx export <id> --with-contents` is accepted and silently behaves as if `--with-content` were absent). Low stakes for a 4-subcommand CLI.
   **Status: Left as-is** for v1; revisit if the flag surface grows.

### Recommendations (reviewer's words, for future passes)

- Reconcile the Go version statement across plans once a decision is made on the 1.25 floor.
- Surface OpenCode's `cost` column in the status table or export output — it's already collected and is more precise than any other provider's data.
- Decide the SQLite connection lifecycle deliberately (cached-forever vs. reconnect-on-Detect-mismatch) rather than leaving it implicit.

### Reviewer's Assessment

**Ready to merge:** With fixes.
**Reasoning (reviewer's words):** "The implementation is faithful to all three plans, integrates cleanly with the Phase 1 interface, and has no critical bugs or broken functionality. The two things genuinely worth addressing before this is 'done' are the `--with-content` arbitrary-out-of-project-path read … and the configprovider double-full-file-read per tick …; everything else is minor or already correct."

Both of those were fixed before this audit was written.

### Fixes applied as a result of this audit

All committed in `7810951` (the three code fixes) and `0fb1b57` (a `.gitignore` correction found while verifying the commit, unrelated to the audit itself but caught in the same pass — a stray `docs/superpowers/*` ignore rule that would have silently stopped future plan docs from being tracked).

1. `internal/export/export.go` — `--with-content` now scoped to the session's working directory; zero-value timestamps omitted instead of rendered as `0001-01-01T00:00:00Z`.
2. `internal/provider/configprovider/provider.go` — `ListSessions` no longer reads file content, eliminating the double full-file read per tick.
3. `cmd/ctx/main.go` — status table skips rows with a zero `LastActive`.
